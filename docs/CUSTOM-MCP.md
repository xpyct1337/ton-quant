# Собственный MCP-сервер: полная инструкция

Как устроен ваш харнес в `mcp_server/`, как писать свои инструменты и как
подключать сервер к Claude Code / Claude Desktop. Обзор **чужих** готовых
серверов — в [MCP-SETUP.md](../MCP-SETUP.md); этот документ про **свой**.

---

## 1. Что такое MCP и зачем свой сервер

**MCP (Model Context Protocol)** — открытый протокол, по которому LLM-клиенты
(Claude Code, Claude Desktop, Cursor и др.) подключают внешние возможности.
Три роли:

- **Host/клиент** — приложение с моделью (Claude Desktop, Claude Code).
- **Сервер** — ваш процесс, который объявляет, что он умеет.
- **Транспорт** — как они общаются: **stdio** (клиент запускает сервер как
  дочерний процесс, обмен JSON-RPC через stdin/stdout — стандарт для локальных
  серверов, наш случай) или **streamable-HTTP** (сервер висит на порту, можно
  подключаться удалённо).

Сервер может отдавать три вида сущностей:

| Сущность | Кто инициирует | Что это |
|---|---|---|
| **Tool** | модель | функция с побочными эффектами/вычислением: «сходи, посчитай, верни» |
| **Resource** | клиент/пользователь | данные только для чтения, адресуемые URI (`harness://about`) |
| **Prompt** | пользователь | переиспользуемый шаблон запроса (в Claude Code видны как slash-команды) |

Зачем свой сервер, а не готовые: готовые дают чужие API, а свой — доступ к
**вашим** данным, скриптам и приватной логике. Одна функция на Python =
инструмент, доступный модели в любом MCP-клиенте.

## 2. Установка окружения

Нужен Python ≥ 3.10 и официальный SDK:

```bash
# вариант А (рекомендуется): uv — ничего не ставит глобально
curl -LsSf https://astral.sh/uv/install.sh | sh   # Windows: winget install astral-sh.uv
cd mcp_server
uv run --with "mcp[cli]" python server.py          # зависимости подтянутся сами

# вариант Б: классический pip
pip install "mcp[cli]"
python mcp_server/server.py
```

`[cli]` добавляет команду `mcp` (Inspector, установка в Claude Desktop).

## 3. Минимальный сервер за 5 минут

Весь работающий сервер — это буквально:

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-harness")

@mcp.tool()
def add(a: int, b: int) -> int:
    """Сложить два числа."""
    return a + b

mcp.run()  # stdio
```

**FastMCP** сам генерирует JSON-схему инструмента из сигнатуры: имя — из имени
функции, описание — из docstring, типы параметров — из аннотаций. Ничего
описывать вручную не нужно — поэтому docstring и аннотации типов обязательны:
это единственное, что модель «видит», выбирая инструмент.

Наш `mcp_server/server.py` — тот же минимум плюс автозагрузка модулей из
`tools/` (см. ниже).

## 4. Анатомия харнеса

```
mcp_server/
├── pyproject.toml       # пакет, зависимость mcp[cli]
├── server.py            # создаёт FastMCP, регистрирует модули, mcp.run()
└── tools/
    ├── __init__.py      # register_all(): импортирует все *.py и зовёт их register(mcp)
    ├── example.py       # образец: tool + resource + prompt
    └── project_data.py  # пример под свои данные: чтение data/*.json
```

> Папка называется `mcp_server/`, а не `mcp/`, намеренно: каталог `mcp/` в
> корне репо затенял бы установленный пакет `mcp` при запуске Python из корня,
> и `from mcp.server.fastmcp import ...` ломался бы.

### Как добавить свой инструмент

1. Скопируйте `tools/example.py` → `tools/моя_задача.py`.
2. Оставьте одну функцию `register(mcp)` и напишите внутри свои инструменты:

```python
def register(mcp) -> None:
    @mcp.tool()
    def my_tool(query: str, limit: int = 10) -> str:
        """Одна строка: что делает инструмент (это читает модель).

        Args:
            query: что искать
            limit: сколько вернуть
        """
        ...
        return result
```

3. Всё. `tools/__init__.py` найдёт файл автоматически — ни списков, ни
   конфигов. Файлы с префиксом `_` игнорируются (черновики).

### Правила, которые сэкономят нервы

- **Типы**: параметры и возврат аннотируйте всегда. Сложные входы — через
  Pydantic-модели (`class Params(BaseModel): ...` как тип параметра) — SDK
  сам построит вложенную схему и провалидирует вход.
- **Ошибки**: поднимайте обычные исключения с понятным текстом
  (`raise ValueError("нет такого файла, есть: ...")`) — клиент получит
  `isError` и текст, модель сможет исправиться и повторить вызов.
- **Размер ответа**: всё, что возвращает инструмент, попадает в контекст
  модели. Большие данные режьте (см. `MAX_CHARS` в `project_data.py`),
  давайте параметры `head`/`keys` для сужения выборки.
- **Секреты** (API-ключи): только через переменные окружения
  (`os.environ["TONAPI_KEY"]`), задаются в блоке `env` конфига клиента.
  Никогда не хардкодьте в коде — репо публичный.
- **Долгие операции**: добавьте параметр `ctx: Context` (из
  `mcp.server.fastmcp`) — получите `await ctx.report_progress(...)` и
  `await ctx.info("лог")`.

## 5. Отладка

### MCP Inspector — главный инструмент

```bash
cd mcp_server
uv run --with "mcp[cli]" mcp dev server.py
```

Откроется веб-интерфейс: видно все tools/resources/prompts, их схемы, можно
вызывать вручную и смотреть сырые JSON-RPC-сообщения. Всегда проверяйте новый
инструмент здесь, прежде чем идти в Claude.

### Типичные грабли

1. **`print()` в инструменте ломает сервер.** stdout занят протоколом —
   любой посторонний вывод рвёт JSON-RPC. Логируйте через `logging`
   (идёт в stderr) или `ctx.info()`.
2. **Клиент не видит новый инструмент** — перезапустите сервер (в Claude
   Desktop — перезапуск приложения; в Claude Code — `/mcp` → reconnect).
   Клиент читает список инструментов при подключении.
3. **Сервер «не запускается» в клиенте** — почти всегда пути/окружение:
   клиент запускает процесс не из вашей папки и не с вашим PATH. Указывайте
   абсолютные пути к интерпретатору и скрипту, проверяйте команду руками
   в чистом терминале.
4. **Относительные пути к данным** — рабочая директория процесса
   непредсказуема; стройте пути от `Path(__file__)` (как в
   `project_data.py`), а не от cwd.

## 6. Подключение к клиентам

### Claude Code (этот репозиторий — уже настроен)

В корне лежит `.mcp.json` — Claude Code подхватывает его автоматически при
работе в репо (спросит подтверждение при первом запуске). Проверка: команда
`/mcp` покажет `ton-quant-harness` и его инструменты.

Вручную/для другого проекта:

```bash
claude mcp add ton-quant-harness -- uv run --with "mcp[cli]" python /abs/path/mcp_server/server.py
```

### Claude Desktop

`%APPDATA%\Claude\claude_desktop_config.json` (Windows) или
`~/Library/Application Support/Claude/claude_desktop_config.json` (macOS),
блок `mcpServers`:

```json
"ton-quant-harness": {
  "command": "uv",
  "args": ["run", "--with", "mcp[cli]", "python",
           "C:\\path\\to\\ton-quant\\mcp_server\\server.py"],
  "env": { "TONAPI_KEY": "ваш-ключ-если-понадобится" }
}
```

Перезапустите Claude Desktop — инструменты появятся в меню 🔌.
Либо одной командой: `uv run --with "mcp[cli]" mcp install server.py`.

### Удалённый вариант (коротко)

Если сервер нужен с других машин: в `server.py` замените запуск на
`mcp.run(transport="streamable-http")` — сервер поднимется на
`http://localhost:8000/mcp`, подключение в клиентах — по URL вместо команды.
Для доступа извне добавьте авторизацию и TLS (reverse-proxy) — без этого
наружу не выставляйте.

## 7. Куда развивать

Идеи следующих модулей `tools/` под ton-quant:

- **`tonapi.py`** — обёртка над TONAPI (цены, jetton-инфо, счета): ключ уже
  есть, паттерн — httpx-запрос + обрезка ответа.
- **`backtest.py`** — запуск `scripts/paper.py` / `xs_forward.py` через
  `subprocess` с возвратом итоговых метрик: «прогони бэктест и покажи PnL»
  одной фразой.
- **`desk.py`** — состояние M1-деска из `data/desk/` (готовый паттерн —
  `project_data.py`).
- Промпты под рутину: «разбери вчерашний снапшот», «сравни сигналы за неделю».

Готовые чужие серверы (Context7, DexScreener, TONAPI и др.) — в
[MCP-SETUP.md](../MCP-SETUP.md); свой харнес и они прекрасно живут вместе.
