# Desk Worker — движок непрерывной работы 24/7 (дизайн)

Дата: 2026-06-28 · Статус: одобрен к реализации (Igor, 28.06) · Версия: v3.0 Phase 1.5
Контекст: [[V3.0-AI-SMART-MONEY-DESK]] (спек деска), [[DESK-SETUP-m1]] (рантайм на M1).

## 1. Цель

Перевести деск с «ночной батч → простой весь день» на **непрерывную работу + само-развитие**
на M1 Air. Главный тезис: **нельзя развиваться без функции приспособленности.** Поэтому
строим замкнутый флайвил «измеряй → улучшай» и стартуем с половины «измеряй»
(**само-калибровка**), а агент-ресёрчер (само-развитие, Phase 2) пристёгивается к ней.

Ночной прогон деска перестаёт быть отдельной сущностью — становится одним из типов задач
в очереди воркера.

## 2. Архитектура

Один долгоживущий процесс под launchd `KeepAlive=true`, никогда не выходит:

```
launchd com.tonquant.worker (KeepAlive=true, RunAtLoad=true)
  └─ scripts/desk_worker.py                ← демон
       ├─ thermal/power governor           ← pmset: жарко/батарея → длинный сон
       ├─ task picker                       ← следующая задача по приоритету
       ├─ task runners (каждый — try, не рушит цикл):
       │    • daily_verdicts               ← новая дата data/ → прогон деска (desk.py)
       │    • backfill_verdicts            ← холодный старт: вердикты за прошлые даты
       │    • calibrate                    ← сверка журналированных вердиктов с исходами
       │    └ (Phase 2) research           ← гипотеза-фактор → CPCV/PBO → выжившие в деск
       └─ batched committer                 ← раз в ~30 мин: pull --rebase → add data/desk → push
```

Старый `com.tonquant.desk` (ночной plist) **выводится** — его роль поглощает воркер
(`daily_verdicts`). `desk.py` / `desk_features.py` переиспользуются как есть.

## 3. Компоненты (юнит = одна задача)

- **`scripts/desk_worker.py`** — демон. *Что:* бесконечный цикл pick→run→pace. *Как:*
  `python3 scripts/desk_worker.py` (launchd) или `--once` (одна итерация, для теста).
  *Зависит:* desk.py, desk_calibration.py, pmset, git.
- **`scripts/desk_calibration.py`** — детерминированная сверка. *Что:* из журнала вердиктов
  + форвардных снапшотов считает, прав ли был деск. *Как:* `build_calibration()` →
  `data/desk/calibration.json`; `--check` self-test. *Зависит:* `data/desk/verdicts/<date>.json`,
  `data/snapshots/*`, форвард-логика из `score.py` (переиспользуем).
- **thermal/power governor** (функция в воркере) — `pmset -g therm` (CPU_Speed_Limit<100 =
  троттлинг) + `pmset -g batt` (AC/battery). Возвращает «работать / спать N сек».
- **batched committer** (функция в воркере) — `git pull --rebase --autostash` → `git add
  data/desk` → commit → push; при rejected — повтор pull+push (до 3 раз), иначе лог и дальше.

## 4. Поток данных и файлы

Проблема: сейчас `verdicts.json` перезаписывается → истории нет, калибровать нечего.

- **Журнал вердиктов:** каждый прогон пишет `data/desk/verdicts/<date>.json` (история) +
  обновляет `data/desk/verdicts.json` (латест — витрина `/desk` читает его как сейчас).
- **Холодный старт = бэкфилл:** есть датированные `data/wallets/*` и `data/snapshots/*` на
  недели назад → воркер прогоняет деск по прошлым срезам, получая backlog вердиктов, у
  которых форвард-исход **уже известен** → калибровка работает с первого дня.
- **Калибровка** → `data/desk/calibration.json` (катящиеся метрики правоты):
  - `high`-risk токены: средняя форвард-доходность (ожидаем хуже/отрицательную);
  - `copy_ok` кошельки: форвард-edge vs отвергнутых (ожидаем выше);
  - hit-rate флагов/порогов по горизонтам +1/+3/+7д.
  Цифры — Python; LLM лишь резюмирует словами (1 вызов на отчёт).
- **Состояние воркера:** `data/desk/worker_state.json` (что сделано, курсор бэкфилла,
  время последнего коммита). Лог: `~/Library/Logs/tonquant-worker.log`.

## 5. PC ↔ Mac координация через git

Три писателя в `main`, **непересекающиеся пути** → ребейз мёржит чисто:
- GitHub Actions (облако) → `data/snapshots/`, `data/wallets/`, и т.п.;
- Mac-воркер → **только** `data/desk/`;
- Windows PC (`deploy.bat`) → `app/`, корневые `*.html`.

Правила воркера:
- коммит **scoped**: `git add data/desk` (не `-A`) — не трогает PC-файлы/мусор;
- перед каждым батч-пушем `pull --rebase --autostash` + ретрай при rejected;
- `.DS_Store` добавляется в `.gitignore` (гигиена 24/7-коммиттера).

Конфликтов на `data/desk/` нет в принципе — туда пишет только Mac. Воркер всегда тянет
свежие данные Actions/PC перед работой.

## 6. Термо/энерго-политика (дефолты)

- Тяжёлая работа **только от сети** (AC). На батарее демон спит (проверка раз в ~5 мин) —
  бережём батарею и тепло пассивного Air.
- При троттлинге (`CPU_Speed_Limit < 100`) — увеличенный сон (back-off).
- Базовый duty-cycle: после задачи — пауза пропорционально нагрузке (работа ≈ пауза),
  чтобы Air не грелся непрерывно. Модель Osaurus остаётся тёплой между итерациями (нет
  cold-reload), но сама выгружается в долгих паузах/на батарее.

## 7. Безопасность само-развития

Phase 1 **только измеряет и репортит**. НЕ меняет пороги/промпты деска автоматически.
Авто-тюнинг и генерация факторов (ресёрчер) = Phase 2, под гейтом CPCV/PBO/DSR + ревью Igor.
Обоснование: 4B не должен молча мутировать торгово-смежную систему; калибровка даёт цифры,
решение об изменениях — человек (пока).

## 8. Отказоустойчивость и тест

- `KeepAlive=true` перезапускает демон при краше; каждый таск в `try` (как деск, «никогда не
  рушим»).
- `desk_worker.py --once` — одна итерация для отладки без демона.
- `desk_calibration.py --check` — self-test (метрики в диапазонах; бэкфилл-вердикты валидны
  по схеме §5; калибровка детерминированно воспроизводима на фикс-срезе).
- Коммиттер не валит цикл при сетевой ошибке пуша (лог + продолжаем; синкнётся позже).

## 9. Phase 2 (вне этого спека, но интерфейс готовим)

Ресёрчер — ещё один task runner: LLM предлагает фактор → Python считает на истории →
CPCV/PBO/DSR гейт → выжившие предлагаются к вливанию в `desk_features.py` (через ревью).
Калибровка из Phase 1 даёт ему сигнал «где деск ошибается = куда копать».

## 10. Вне scope (YAGNI)

- Новые on-chain фетчи (форензика transfer-таймлайна) — отдельный спек.
- Авто-мутация порогов/промптов — Phase 2.
- Веб-управление воркером с сайта — нет канала записи (как в `/desk` note).
- Распараллеливание агентов — 8 ГБ, один в памяти.
